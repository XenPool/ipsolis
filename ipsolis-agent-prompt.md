## CRITICAL BUGS — Fix first

### B1 · Language preference not persisted across server-rendered pages
**Symptom:** Switching language to EN in the portal, then navigating to `/portal/orders/238` or `/portal/delegations` keeps the EN locale even after switching back to DE. The language cookie/session value appears not to be read server-side on every render.
**Fix:** Ensure the language preference (cookie or session key, e.g. `lang`) is read in every SSR route handler / Jinja base template context. The `Accept-Language` header or session cookie must be evaluated on each request, not only on the language-toggle POST endpoint.
**Files to check:** base template (e.g. `templates/base.html`), language middleware, session/cookie read logic, all portal route handlers.

### B2 · Execution step duration shows `-0.0s` (negative zero)
**Symptom:** Every completed execution step in both portal and admin order detail views shows `Duration: -0.0s`.
**Fix:** The duration is calculated as `end_time - start_time`. The result is being stored or rounded to a negative zero float. Apply `max(0, duration)` before formatting. Display as `< 1 s` if the rounded value is 0, or omit the duration label entirely if it is zero.
**Files to check:** The step serializer / Pydantic model for order steps, the template that renders `Dauer:` / `Duration:`, and the Celery task that records timing.

### B3 · Requester display name inconsistent: "Stefan" vs. "Stefan van Boxmer-Fischer"
**Symptom:** Orders #237, #227, #226 show only first name "Stefan" in the orders list; newer orders show the full name. This is a data inconsistency that may indicate the display name is constructed differently across code paths.
**Fix:** Audit the order creation logic and the field used to store/render the requester's display name. Ensure a single canonical field (e.g. `display_name` or `full_name`) is populated at order creation time and used consistently. If legacy orders lack the field, add a data migration that back-fills from AD or from the email's linked user record.
**Files to check:** Order creation endpoint, User model, order list template / serializer.

## HIGH PRIORITY — Admin Portal

### A1 · Admin sidebar nav link "Runbooks" points to wrong URL slug
**Symptom:** The sidebar label is "Runbooks" but navigating directly to `/ui/runbooks` returns `{"detail":"Not Found"}` (raw JSON 404). The actual route is `/ui/standalone-runbooks`.

### A2 · Route `/ui/asset-definitions` returns raw JSON 404 instead of HTML error page
**Symptom:** Navigating to `/ui/asset-definitions` (a plausible URL, shown as the sidebar label "Asset Definitions") returns `{"detail":"Not Found"}` because the actual route is `/ui/asset-types`.
**Fix:** Add a 301/302 redirect from `/ui/asset-definitions` → `/ui/asset-types`, OR add a proper HTML 404 page handler for unknown `/ui/*` routes so users never see a raw JSON response in the browser.
**Files to check:** Admin router, global 404 exception handler for the `/ui` prefix.

### A3 · Global 404 handler for `/ui/*` routes missing (raw JSON shown in browser)
**Symptom:** Any unknown `/ui/` path (e.g. `/ui/runbooks`, `/ui/asset-definitions`) renders `{"detail":"Not Found"}` — the FastAPI default JSON error — directly in the browser. This is a backend API JSON error leaking into the SPA/SSR layer.
**Fix:** Add an exception handler or catch-all route for `/ui/*` that returns a styled HTML 404 page (using the base admin template). The JSON 404 handler is correct for `/api/*` endpoints but must not apply to browser-facing UI routes.
**Files to check:** Main FastAPI app setup, exception handlers, admin router.

### A4 · Order detail in admin: status badge `pending_approval` has no colour / underscore in value
**Symptom:** Orders #229 and #228 show `pending_approval` as a plain unstyled text badge. The underscore makes it look like a raw DB enum value.
**Fix:** (a) Add a CSS badge style for `pending_approval` (suggest amber/yellow, consistent with "waiting" semantics). (b) Render the display string as "Pending Approval" (title-case, no underscore) everywhere badges appear.
**Files to check:** Badge CSS/Tailwind utility map, order status display helper, all templates rendering order status.

### A5 · Order detail admin: rejected orders show "Error" box with approval comment
**Symptom:** Order #236 (rejected) shows an "Error" box with the text `Declined by Alice: no thanks`. A rejection is not an error — using the red "Error" component is semantically wrong and alarming.
**Fix:** Add a separate "Rejection reason" display component (e.g. amber/neutral info box with label "Rejection reason:") that is shown when `status == rejected`. Reserve the red "Error" box strictly for technical failures.
**Files to check:** Admin order detail template, status-conditional rendering logic.

### A6 · Settings page: Update Notifier shows persistent error inline next to Save button
**Symptom:** On `/ui/settings`, the Update Notifier section shows `Last check: 2026-04-27… · Error: HTTPError: HTTP Error 401: Unauthorized` as inline red text next to the Save button. This is a dev-environment credential issue, but the error display is poorly placed and uses raw exception text.
**Fix:** (a) Move the last-check status to a separate read-only status line below the form, not beside the Save button. (b) Format the error as a user-friendly message: "Last check failed — GitHub token may be missing or invalid." Log the raw exception server-side only.
**Files to check:** Settings template, Update Notifier background task error handling.

### A7 · Admin login placeholder text truncated — too long for input field

### A8 · Admin portal has no language selector (login page is always English)
